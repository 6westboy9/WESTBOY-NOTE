## 三级模式-两级映像

**三级模式：**

- **外模式**：<font color="#c00000">用户看到的数据视图</font>。比如学生只能看到自己的成绩，老师能看到全班成绩，不同用户看到的数据范围可以不同。
- **概念模式**：<font color="#c00000">数据库的全局逻辑结构</font>。比如有哪些表、字段、关系、约束等，是整个数据库的统一逻辑视图。
- **内模式**：<font color="#c00000">数据在磁盘上的物理存储方式</font>。比如文件组织方式、索引结构、存储路径等。

**两级映像：**

- **外模式 / 概念模式映像**：描述用户视图和全局逻辑结构之间的对应关系。它保证**逻辑数据独立性**，即概念模式改了，尽量不影响用户视图和应用程序。
- **概念模式 / 内模式映像**：描述全局逻辑结构和物理存储结构之间的对应关系。它保证**物理数据独立性**，即底层存储方式改了，尽量不影响数据库逻辑结构和应用程序。

```
外模式：用户看到什么
概念模式：数据库逻辑上是什么
内模式：数据物理上怎么存

外模式/概念模式映像 -> 逻辑数据独立性
概念模式/内模式映像 -> 物理数据独立性
```

## 数据库设计

- 数据库设计一般包括六个阶段：需求分析、概念结构设计、逻辑结构设计、物理设计、数据库实施、数据库运行和维护。

- <font color="#c00000">需求分析</font>：分析数据存储的要求，产出物通常包括数据流图、数据字典、需求说明书。
	- 需要获得用户对系统的三个要求：信息要求、处理要求、系统要求。
- <font color="#c00000">概念结构设计</font>：设计 E-R 图，也就是实体-联系图。
	- 工作步骤：选择局部应用、逐一设计分 E-R 图、合并 E-R 图。
	- 分 E-R 图合并时，主要冲突有三类：
		- 属性冲突：同一属性可能存在于不同的分 E-R 图中。
		- 命名冲突：相同意义的属性在不同分 E-R 图中命名不同；或名称相同的属性在不同分 E-R 图中代表不同意义。
		- 结构冲突：同一实体在不同分 E-R 图中有不同属性；或同一对象在某一分 E-R 图中被抽象为实体，在另一分 E-R 图中又被抽象为属性。
- <font color="#c00000">逻辑结构设计</font>：将 E-R 图转换成关系模式。
	- 工作步骤：确定数据模型、将 E-R 图转换为指定的数据模型、确定完整性约束、确定用户视图。
- <font color="#c00000">物理设计</font>：确定数据分布、存储结构和访问方式。
- <font color="#c00000">数据库实施阶段</font>：根据逻辑设计和物理设计阶段的结果建立数据库，编制与调试应用程序，组织数据入库，并进行试运行。
- <font color="#c00000">数据库运行和维护阶段</font>：数据库应用系统经过试运行即可投入运行，但该阶段需要不断对系统进行评价、调整与修改。

## 数据模型

- <font color="#c00000">关系模型</font>是用二维表的形式表示实体-联系模型，是由实体-联系模型转换而来，并经过开发人员设计后的数据模型。
- <font color="#c00000">概念模型</font>是从用户角度进行建模的，是现实世界到信息世界的第一抽象，也是真正的实体-联系模型。

- 常见数据模型：
	- 关系模型：用二维表表示实体及实体之间的联系。
	- 网状模型：表示实体类型及其实体之间的联系，一个事物可以和另外多个事物存在联系，形成一张网。
	- 面向对象模型：采用面向对象的方法设计数据库，以对象为单位，每个对象包括属性和方法，具有类和继承等特点。

- <font color="#c00000">数据模型三要素</font>：
	- 数据结构：所研究的对象类型的集合。
	- 数据操作：对数据库中各种对象的实例允许执行的操作的集合。
	- 数据的约束条件：一组完整性规则的集合。

- E-R 图用于描述概念数据模型。现实世界由一组称为实体的基本对象，以及这些对象之间的联系构成。

- E-R 模型中的常见表示方法：
	- 椭圆表示属性。
	- 长方形表示实体。
	- 菱形表示联系。
	- 联系的两端要填写联系类型。

# 规范化和并发控制

## 函数依赖

- 给定一个 `X`，能唯一确定一个 `Y`，就称 `X` 确定 `Y`，或者说 `Y` 依赖于 `X`，记作 `X -> Y`。
	- 例如：`Y = X * X` 时，给定 `X` 的值就能唯一确定 `Y` 的值。

- 函数依赖可以扩展出以下两种常见规则：
	- 部分函数依赖：`A` 可以确定 `C`，`(A, B)` 也可以确定 `C`，但 `(A, B)` 中的一部分，即 `A`，已经可以确定 `C`，此时称为部分函数依赖。
	- 传递函数依赖：当 `A` 和 `B` 不等价时，若 `A` 可以确定 `B`，`B` 可以确定 `C`，则 `A` 可以确定 `C`，此时称为传递函数依赖；若 `A` 和 `B` 等价，则不存在传递，直接就可以确定 `C`。

- 函数依赖的公理系统也称为 Armstrong 公理系统。

- 设关系模式 `R<U, F>`，`U` 是关系模式 `R` 的属性全集，`F` 是关系模式 `R` 的一个函数依赖集。对于 `R<U, F>`，有以下推理规则：
	- 自反律：若 `Y ⊆ X ⊆ U`，则 `X -> Y` 为 `F` 所逻辑蕴含。
	- 增广律：若 `X -> Y` 为 `F` 所逻辑蕴含，且 `Z ⊆ U`，则 `XZ -> YZ` 为 `F` 所逻辑蕴含。
	- 传递律：若 `X -> Y` 和 `Y -> Z` 为 `F` 所逻辑蕴含，则 `X -> Z` 为 `F` 所逻辑蕴含。
	- 合并规则：若 `X -> Y`，`X -> Z`，则 `X -> YZ` 为 `F` 所蕴涵。
	- 伪传递律：若 `X -> Y`，`WY -> Z`，则 `XW -> Z` 为 `F` 所蕴涵。
	- 分解规则：若 `X -> Y`，`Z ⊆ Y`，则 `X -> Z` 为 `F` 所蕴涵。

## 键与约束

- <font color="#c00000">超键</font>：能<u>唯一标识</u>此表的属性组合。
	- 例如：学生表中，`学号` 能唯一标识一名学生，`学号 + 姓名` 也能唯一标识一名学生，因此二者都可以是超键。
- <font color="#c00000">候选键</font>：<u>超键中去掉冗余属性后</u>，剩余的属性就是候选键。
	- 例如：`学号 + 姓名` 是超键，但 `姓名` 是冗余属性，去掉后 `学号` 仍能唯一标识学生，所以 `学号` 是候选键。
- <font color="#c00000">主键</font>：<u>任意一个候选键都可以作为主键</u>。
	- 例如：学生表中如果 `学号` 和 `身份证号` 都是候选键，可以任选一个作为主键，通常选择 `学号`。
- <font color="#c00000">外键</font>：<u>其他表中的主键</u>。
	- 例如：成绩表中的 `学号` 引用学生表的主键 `学号`，则成绩表中的 `学号` 是外键。
- <font color="#c00000">主属性</font>：候选键内的属性为主属性，其他属性为非主属性。
	- 例如：若候选键是 `学号`，则 `学号` 是主属性，`姓名、年龄、专业` 等是非主属性。
- 实体完整性约束：即主键约束，主键值不能为空，也不能重复。
- 参照完整性约束：即外键约束，外键必须是其他表中已经存在的主键的值，或者为空。
- 用户自定义完整性约束：自定义表达式约束，如设定年龄属性的值必须在 `0~150` 之间。
