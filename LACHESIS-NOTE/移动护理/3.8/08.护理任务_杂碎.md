# 护理任务问题案例：`TaskRemindRecord.wardCode` 缺失导致 `interceptBedCode` 为空

> [!important] 一句话结论
> 当护理任务记录上的 `wardCode` 缺失、为空或错误时，患者当前信息上的 `bedCode` 仍可能正常返回，但 `interceptBedCode` 会因为缺少病区白板配置上下文而无法计算。

> [!note] 当前文档定位
> 本文先沉淀一个已经确认根因的护理任务问题案例，后续护理任务相关的其它零散问题，也可以继续按“场景 -> 链路 -> 排查 -> 修复 -> 验证”的结构补充在本文件中。

## 1. 业务场景

护理任务列表接口：

`GET /api/windranger/Mnis/taskReminds/list`

排查这个问题时，前端最直观的异常现象是：

- 接口返回的 `queryResult` 中，`bedCode` 有值
- 同一条数据中的 `interceptBedCode` 却为 `null`

这个问题表面上看像“床号截断失败”，但实际并不是床号规则本身算错，而是**计算截床号时缺少病区上下文**。

## 2. 接口与核心方法入口

### 接口入口

- 网关路径：`GET /api/windranger/Mnis/taskReminds/list`

### 核心方法入口

- 护理任务查询与结果补充：`TaskRemindService.filterTaskRemind(...)`
- 截床号计算：`patientUtils.addInterceptBedCode(...)`

### 这个问题发生在哪一步

本问题不是任务生成页面报错，也不是患者基础信息查询失败，而是**护理任务列表返回前，对结果进行二次补充时**出现字段不一致。

## 3. 关键字段来源对照

| 字段 | 用途 | 实际来源 | 是否依赖病区配置 |
| --- | --- | --- | --- |
| `bedCode` | 展示患者原始床号 | 当前患者信息 `PatInhosRecord` | 否 |
| `interceptBedCode` | 展示截断后的床号 | `TaskRemindRecord.wardCode + WhiteBoardConfig` 二次计算 | 是 |
| `wardCode` | 作为病区上下文参与截床号计算 | 护理任务记录 `TaskRemindRecord` | 是 |
| `WhiteBoardConfig.config.basicConfig` | 提供床号截断规则 | 病区白板配置 | 是 |

这几个字段的关键差异在于：

- `bedCode` 取的是当前患者信息
- `interceptBedCode` 取的是任务记录上的 `wardCode` 再结合白板配置计算

也就是说，这两个字段虽然最终都和“床号展示”有关，但**来源并不一致**。

## 4. 现象与直接判断

线上排查时发现：

- 如果返回元素上没有 `wardCode`，`interceptBedCode` 算不出来
- 如果临时给该元素补上正确的 `wardCode`，再执行截床号逻辑，`interceptBedCode` 就可以正常生成

因此可以直接得到一个初步判断：

> 这类问题首先应该怀疑 `TaskRemindRecord.wardCode` 是否缺失，而不是先怀疑床号截断正则。

## 5. 查询链路

在 `TaskRemindService.filterTaskRemind(...)` 中，整体处理顺序可以理解为：

1. 先从任务记录中取 `wardCode`
2. 将 `TaskRemindRecord` 转成 `TaskRemindRecordExt`
3. 再从患者信息中补 `bedCode`、`patName` 等字段
4. 最后按任务记录中的 `wardCode` 分组，调用 `patientUtils.addInterceptBedCode(...)` 计算 `interceptBedCode`

这里最容易误判的地方是：

- `bedCode` 正常，不代表参与截床号计算的上下文也正常
- `interceptBedCode` 是否能算出，取决于任务记录上的 `wardCode` 是否有效

因此，出现下面这种组合并不矛盾：

- `bedCode` 正常
- `interceptBedCode` 为空

根本原因就是：**查询阶段拼装结果时，两个字段取值链路不同。**

## 6. 为什么补了 `wardCode` 就能算出 `interceptBedCode`

`patientUtils.addInterceptBedCode(wardCode, list)` 能生效，前提至少包括：

- 能根据 `wardCode` 查到对应病区的 `WhiteBoardConfig`
- `WhiteBoardConfig.config.basicConfig` 存在

如果 `wardCode` 为空、缺字段或者不是正确病区，方法通常会直接返回，不会给对象补上 `interceptBedCode`。

所以这类现象的本质是：

- 原始 `bedCode` 是有的
- 但缺少“按哪个病区规则截取床号”的上下文

## 7. 为什么不是床号规则本身的问题

`processBedCode(...)` 的逻辑特征是：

- 如果 `bedCodeRegex` 为空，直接返回原始 `bedCode`
- 如果正则不匹配，也直接返回原始 `bedCode`

也就是说，正常情况下它不会主动把一个原本存在的床号处理成 `null`。

因此，当你看到：

- `bedCode` 有值
- `interceptBedCode` 为空

更优先应该排查：

- `wardCode` 缺失
- `wardCode` 错误
- 查询兼容进了历史脏数据

而不是优先排查床号截断正则本身。

## 8. 当前已确认的高风险来源

### 8.1 历史数据本身没有 `wardCode`

查询任务时，代码显式兼容了“没有 `wardCode` 字段”的旧数据。

这意味着即使当前请求已经带了病区，也仍然可能把旧任务一起查出来。这类历史数据，是最直接的异常来源。

### 8.2 护理等级触发的任务写入时没有补 `wardCode`

`NurseLevelTaskRemindHandler` 会直接保存 `assembleTaskRemindRecord(...)` 生成的任务。

但其父类 `AbstractTaskRemindRecordHandler.buildTaskRemindRecord(...)` 中，没有给 `TaskRemindRecord` 赋 `wardCode`。

这条链路生成的新任务，存在 `wardCode` 为空的风险。

### 8.3 长期任务拆分后的当天任务写入时没有补 `wardCode`

`TaskRemindSplitService.createAndSaveCurrentDayTasks(...)` 会直接保存 `taskRemindUtils.constructTaskRemindRecords(...)` 生成的任务。

而 `constructTaskRemindRecords(...)` 依赖的 `initOrderTaskRemind(...)` 本身没有设置 `wardCode`。

因此，这条链路也会生成 `wardCode` 为空的任务。

### 8.4 已有脏任务被复制后继续传播

`FirstDayUnderHandler` 在恢复任务时，是基于原 `TaskRemindRecord` 模板复制新对象。

如果模板任务本身 `wardCode` 就为空，那么复制出来的新任务也会继续为空。

## 9. 当前仍需单独标记为“推测”的部分

下面这些判断方向目前是合理推断，但建议后续结合实际数据再补充证据：

- `wardCode = ''` 更像历史脏数据残留，或者上游传入空值后被直接保存
- 除了已列出的几条链路外，可能还存在其它直接 `new TaskRemindRecord()` 并落库的写入入口

为了避免“推测”和“已确认”混在一起，后续补充内容时建议继续保持这两个层级分开记录。

## 10. 当前临时修复思路

可以先在 `filterTaskRemind(...)` 中，对“按任务记录分组补 `interceptBedCode`”的逻辑做读侧兜底：

```java
recordMap.forEach((wCode, recordExts) -> {
    if (isBlank(wCode)) {
        patientUtils.addInterceptBedCode(wardCode, recordExts);
    } else {
        patientUtils.addInterceptBedCode(wCode, recordExts);
    }
});
```

这个修复的意义是：

- 当任务记录自身 `wardCode` 为空时，回退使用当前查询病区 `wardCode`
- 能直接修复接口返回中 `interceptBedCode = null` 的展示问题

但要注意：

- 这是读侧止血
- 不是写侧根修
- 新脏数据仍然可能继续生成

## 11. 更稳的根治方向

### 11.1 读侧统一来源

不要再让：

- `bedCode` 取当前患者信息
- `interceptBedCode` 取任务记录病区后二次计算

更稳的做法是：

- 直接复用患者信息上已计算好的 `interceptBedCode`
- 避免两套来源不一致

### 11.2 写侧补齐 `wardCode`

重点检查并修复以下入库链路：

- `NurseLevelTaskRemindHandler`
- `TaskRemindSplitService`
- 其他所有直接 `new TaskRemindRecord()` 并保存的链路

治理目标是：

- 新生成的护理任务都必须带 `wardCode`

### 11.3 必要时回补历史脏数据

如果线上历史脏数据较多，可以考虑批量回补 `TaskRemindRecord.wardCode`。

但这里不能简单做“按当前患者病区全量回填”，因为要考虑患者转科等场景。至少需要结合：

- 当前患者病区
- 任务创建时间
- 任务触发来源

综合判断后再回补。

## 12. 建议的排查步骤

### 12.1 快速判断是不是本问题

如果同一条护理任务数据中出现下面这个组合：

- `bedCode` 有值
- `interceptBedCode` 为 `null`

就优先按本案例方向排查。

### 12.2 推荐排查顺序

1. 先看接口返回，确认异常数据是否存在 `wardCode`
2. 再看对应任务记录，确认落库时 `wardCode` 是否缺失或为空
3. 再确认该病区是否能查到 `WhiteBoardConfig.config.basicConfig`
4. 再回溯任务的生成来源，判断是历史脏数据还是新链路持续产出脏数据
5. 最后再看是否需要做读侧兜底、写侧修复或历史数据回补

### 12.3 排查时建议重点区分的两类情况

- 历史脏数据：旧任务没有 `wardCode`，但新任务未必还会继续产生
- 持续新增脏数据：新的写入链路仍在生成没有 `wardCode` 的任务

这两类情况的治理策略不同，不建议混在一起处理。

## 13. 修复后的验证闭环

### 13.1 验证读侧兜底是否生效

修复后重新调用：

`GET /api/windranger/Mnis/taskReminds/list`

需要重点确认：

- 原来 `interceptBedCode = null` 的记录，是否已经能正常返回截床号
- 相同病区下的其它任务展示是否被误伤
- 不同病区查询时，是否仍按各自病区规则截床号

### 13.2 验证写侧是否已根修

针对护理等级任务、长期任务拆分任务、恢复任务等场景，分别确认：

- 新生成的 `TaskRemindRecord` 是否都带 `wardCode`
- 不再依赖读侧兜底也能算出 `interceptBedCode`

### 13.3 验证历史脏数据治理是否安全

如果做了历史数据回补，还需要重点确认：

- 转科患者有没有被错误回填到当前病区
- 同一患者不同时间生成的任务，病区归属是否仍然正确

## 14. 影响范围判断

这个问题的影响更偏向“护理任务列表展示与读侧字段补充”，具体表现为：

- 患者原始床号 `bedCode` 可能正常
- 截床号 `interceptBedCode` 为空
- 同一患者在不同来源任务上的展示效果可能不一致

当前内容没有证据表明它会直接影响患者基础信息本身，但会影响护理任务列表的展示一致性，以及基于病区规则的床号呈现。

## 16. 一句话总结

护理任务中 `TaskRemindRecord.wardCode` 为空时，最直接的表现就是：

- 当前患者能查到 `bedCode`
- 但由于缺少病区上下文，无法基于白板配置计算 `interceptBedCode`

这次问题的本质是：

**任务记录病区信息缺失，导致读侧床号展示计算失败。**
