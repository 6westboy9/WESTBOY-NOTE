npx 是 npm 从 v5.2.0 开始内置的「包执行器」，专门用来快速运行 Node.js 包/命令，不用手动全局安装，也不会污染本地环境。

简单说：

- **npm**：用来安装/管理包（装软件）
- **npx**：用来直接运行包（用完即走）

---

## 一、npx 的诞生背景与前生今世

### 1. 前世：没有 npx 的时代（2017 年之前）

在 npm v5.2.0 之前，你想运行一个命令行工具（比如 `create-react-app`、`vue-cli`、`eslint`），必须先全局安装：

```bash
# 必须先全局安装
npm install -g create-react-app

# 安装完才能用
create-react-app my-app
```

这会带来一堆痛点：

1. **全局污染**：装一堆全局包，占空间、难管理
2. **版本冲突**：A 项目要 v1 版，B 项目要 v2 版，全局只能留一个
3. **步骤繁琐**：想临时用个工具，必须先装、用完还得删
4. **权限问题**：全局安装经常报 `permission denied` 错误

### 2. 诞生：2017 年，npm v5.2.0 内置 npx

npm 团队为了解决上面所有问题，推出了 npx：

- 无需全局安装
- 直接运行命令
- 自动缓存、自动清理
- 支持临时运行、支持最新版、支持指定版本

它不是独立工具，就是 npm 的一部分，装了 Node.js 就自带 npx。

### 3. 今世：现在的标准开发习惯

现在前端/Node 开发，90% 的临时命令都用 npx 执行，已经成为行业标准：

```bash
npx create-react-app my-app
npx vue create my-project
npx vite
npx eslint .
```

## 二、没有 npx 之前 VS 有了 npx 之后

### 没有 npx

```bash
# 步骤 1：全局安装（必须）
npm install -g 某个包

# 步骤 2：才能运行
包名 命令

# 步骤 3：用完想卸载还要手动删
npm uninstall -g 包名
```

- 麻烦、污染环境、版本冲突、权限报错

### 有了 npx

```bash
# 一步到位：直接运行，不用安装
npx 包名 命令
```

- 自动下载 → 自动运行 → 用完不残留（或缓存加速下次）
- 不污染全局、无版本冲突、零成本试用工具

---

## 三、npx 最核心的能力

1. **不用全局安装，直接运行命令行包**
2. **运行本地项目依赖里的命令**（不用写 npm scripts）
3. **直接运行指定版本**：`npx 包名@版本号`
4. **直接运行 GitHub gist / 远程脚本**
5. **运行一次性工具，用完即走**

---

## 四、npx 的优点

1. **零污染全局环境**

   不用再装一堆 `-g` 全局包，电脑更干净。

2. **永远使用最新版**

   不用手动更新全局包，npx 默认跑最新版。

3. **支持指定版本，无冲突**

   ```bash
   npx create-react-app@4
   npx create-react-app@5
   ```

   不同版本互不干扰。

4. **简化流程**

   从「安装 → 运行 → 删除」变成直接运行。

5. **解决本地依赖命令调用问题**

   项目里装了 `eslint`，以前只能在 `package.json` 里写脚本，现在直接：`npx eslint .`

6. **无权限问题**

   不用 `sudo` / 管理员权限，安全又方便。

---

## 五、npx 的缺点

1. **第一次运行需要下载**

   首次使用会慢一点，第二次会缓存，速度正常。

2. **不适合高频固定工具**

   如果你每天都用某个包，全局安装还是更快。

3. **必须联网**

   离线环境无法自动下载包，无法使用。

4. **缓存可能过时**

   极少数情况缓存没更新，可加 `--no-install` 强制重新下载。

---

## 六、总结

- **npx = npm 的执行工具**，npm 管安装，npx 管运行
- **诞生前**：必须全局安装，污染环境、版本冲突、步骤繁琐
- **诞生后**：直接运行、用完即走、无全局污染、无版本冲突
- **优点**：干净、方便、安全、最新版、无冲突
- **缺点**：首次稍慢、需要联网、不适合超高频固定工具

**一句话记忆：想用什么 Node 命令，直接前面加 npx，不用装，直接跑！**
